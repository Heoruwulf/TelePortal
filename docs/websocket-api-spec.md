# TelePortal WebSocket API Specification

This document details the WebSocket interface provided by TelePortal for integrating AI agents.

## Overview

TelePortal bridges standard SIP/RTP telephony to a WebSocket connection. Once a SIP `INVITE` is accepted, TelePortal establishes an audio bridge. Your AI agent connects to this bridge via WebSocket to send and receive real-time audio and control signals (like DTMF).

* **Audio Data:** Sent and received as **Binary** WebSocket frames.
* **Control Data:** Sent and received as **Text (JSON)** WebSocket frames.

## 1. Connection Lifecycle

### 1.1 Redis Pub/Sub Notification (Call Events)

TelePortal uses Redis Pub/Sub to broadcast real-time lifecycle events. Orchestrators and AI agents should subscribe to the following channel to receive notifications about new calls:

* **Channel Name:** `teleportal.calls.events`

When a SIP call is successfully established (connected), TelePortal publishes a `CallEvent` JSON message to this channel.

**Example Notification (`connected` event):**

```json
{
  "event": "connected",
  "call_id": "sip-call-id-1234",
  "internal_id": "uuid-v4-5678",
  "websocket_url": "ws://teleportal.example.com/v1/listen/uuid-v4-5678/sip-call-id-1234",
  "metadata": {
    "From": "sip:+15551234567@provider.com",
    "To": "sip:agent@teleportal.example.com",
    "User-Agent": "TelePortal-v1.0"
  },
  "timestamp": "2026-05-17T10:00:00.123456Z"
}
```

| Field | Type | Description |
| :--- | :--- | :--- |
| `event` | string | The lifecycle event. Common values: `connected`, `disconnected`. |
| `call_id` | string | The original SIP Call-ID. |
| `internal_id` | string | A unique UUID generated by TelePortal for this specific session. |
| `websocket_url` | string | **[CRITICAL]** The full URL your agent must connect to. |
| `metadata` | map | A key-value map of the SIP headers from the original `INVITE`. |
| `timestamp` | string | RFC3339 formatted timestamp of the event. |

**Lifecycle Note:**

* **`connected`**: Published when the SIP 200 OK is sent. The WebSocket bridge is now ready for connection.
* **`disconnected`**: Published when the call ends (BYE or CANCEL). Any active WebSocket connection will be closed shortly after this event.

### 1.2 Establishing the WebSocket

Your AI agent connects to the `websocket_url` provided in the event.

* **Endpoint Format:** `ws://<host>:<port>/v1/listen/:internal_id/:call_id`
* **Authentication:** If `TELEPORTAL_AUTH_JWT_SECRET` is set on the server, you **must** include an Authorization header:
  * `Authorization: Bearer <your_jwt_token>`

### 1.3 Initial Handshake (Server to Client)

Immediately upon connecting, TelePortal will send two JSON text frames to initialize the session:

**Message 1: Call Headers**
Contains the SIP headers from the original `INVITE` request.

```json
{
  "type": "headers",
  "headers": {
    "From": "sip:+15551234567@provider.com",
    "To": "sip:agent@teleportal.example.com"
  },
  "started": "2026-05-17T10:00:00Z"
}
```

**Message 2: Audio Metadata**
Contains critical information about how to decode the incoming binary audio and how you must encode your outgoing audio.

```json
{
  "type": "metadata",
  "codec": "L16",
  "sample_rate": 8000,
  "channels": 1,
  "ptime": 20,
  "dtmf_enabled": true,
  "is_big_endian": false
}
```

## 2. Audio Processing (Binary Frames)

Audio is streamed exclusively over **Binary** frames.

### 2.1 Audio Codecs

The codec used on the WebSocket is determined by the `TELEPORTAL_WS_CODEC` environment variable on the server.

* **L16 (Recommended for AI):** 16-bit Linear PCM. Uncompressed, raw audio. Ideal for immediate ingestion into Speech-to-Text engines.
  * *Endianness:* Little-Endian by default. (If `PASS` mode is used and L16 is negotiated via SIP, it will be Big-Endian. Check the `is_big_endian` flag in the metadata message).
* **PCMU / PCMA:** G.711 mu-law or A-law. 8-bit companded audio. Requires decoding on your end before AI processing, but uses half the bandwidth of L16.

### 2.2 Streaming Mechanics

* **Receiving (Server to Client):** TelePortal sends audio chunks continuously as they are received and processed from the RTP stream.
* **Sending (Client to Server):** Your agent should stream binary audio back to TelePortal.
  * **Pacing:** You should aim to send audio in chunks that match the `ptime` (packet time, usually 20ms). Sending massive chunks at once can cause buffer overflows or increased latency on the SIP side.
  * **Format:** The audio you send **must** exactly match the format specified in the initial `metadata` message (e.g., L16, 8000Hz, Mono).

## 3. Control Messages (Text/JSON Frames)

All control signaling happens via JSON text frames.

### 3.1 Receiving DTMF (Caller Pressed a Key)

If the caller presses a number on their keypad, TelePortal detects the RFC 4733 event and forwards it to your agent.

**Server to Client:**

```json
{
  "type": "dtmf",
  "digit": "5",
  "duration": 160,
  "ts": 1715684400000
}
```

### 3.2 Sending DTMF (Agent Presses a Key)

Your AI agent can simulate key presses (e.g., to navigate external IVR systems).

**Client to Server:**

```json
{
  "type": "dtmf",
  "digit": "9",
  "duration": 200
}
```

*Note: `duration` is in milliseconds.*

### 3.3 Terminating the Call

**Agent Initiated (Client to Server):**
To hang up the call, send a `bye` request. TelePortal will close the WebSocket and send a SIP `BYE` to the caller.

```json
{
  "type": "bye"
}
```

**Caller Initiated (Server to Client):**
If the caller hangs up first, TelePortal will notify your agent before closing the WebSocket connection.

```json
{
  "type": "call_ended",
  "call_ended": true,
  "ts": 1715684500000
}
```

## 4. Best Practices for AI Integration

1. **Use the Go Client:** If your agent is written in Go, use the provided `pkg/client`. It handles the WebSocket lifecycle, automatic reconnections, and strongly-typed JSON parsing for you.
2. **Voice Activity Detection (VAD):** TelePortal streams *continuous* audio, including silence. Your AI agent should implement VAD to determine when the user is speaking before sending data to an STT API to save costs.
3. **Handle Disconnects:** If your WebSocket disconnects unexpectedly, the SIP call is still active. Reconnect immediately using the same URL.
4. **Barge-in (Interrupts):** If the AI is speaking and the user interrupts, your agent must stop sending binary audio frames immediately. TelePortal simply forwards what you send; it does not implement echo cancellation or barge-in logic natively.
